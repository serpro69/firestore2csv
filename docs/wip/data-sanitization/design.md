# Design: Data Sanitization

## Overview

Data sanitization replaces PII (personally identifiable information) in exported Firestore data with realistic fake values generated by the `gofakeit` library. It is available in two forms:

1. **`--sanitize` flag on the `export` subcommand** — opt-in, replaces PII during export before CSV is written.
2. **`sanitize` subcommand** — standalone CSV-in/CSV-out transformation for sanitizing previously exported files.

Both share the same sanitization engine and configuration format.

## Sanitization Config

### YAML file (flat structure)

```yaml
fields:
  email: email
  first_name: firstName
  last_name: lastName
  company: companyName
  phone: phone
  address: address
  user_id: uuid
```

### Inline format

Comma-separated `key=type` pairs:

```
email=email,first_name=firstName,phone=phone
```

### Supported faker types

| Config type    | Description              | gofakeit function       |
|---------------|--------------------------|------------------------|
| `firstName`   | First name               | `FirstName()`          |
| `lastName`    | Last name                | `LastName()`           |
| `email`       | Email address            | `Email()`              |
| `phone`       | Phone number             | `Phone()`              |
| `address`     | Full street address      | `Address().Address`    |
| `companyName` | Company name             | `Company()`            |
| `uuid`        | UUID string              | `UUID()`               |

## Field Matching

- **Exact key name matching** — a field is sanitized only if its name exactly matches a key in the config.
- **Global across all collections** — the same rules apply to every collection. No per-collection scoping.
- **Recursive into nested structures** — the sanitizer walks nested `map[string]any` values (maps) and arrays of maps, matching field names at every depth level.
- **String values only** — if a matched field holds a non-string value (e.g., an integer), it is left untouched. Sanitization only replaces string values.

## Seed Control

- A `--seed` flag (int64) controls randomness.
- Default value `0` means fully random (non-deterministic across runs).
- Any non-zero seed produces deterministic output — the same input always yields the same fake value for a given seed.
- The seed is passed to `gofakeit.New(seed)` to create a seeded faker instance.

## CLI Interface

### Export with sanitization

```bash
# Inline config
firestore2csv export -p myproject --sanitize "email=email,name=firstName"

# YAML config file
firestore2csv export -p myproject --sanitize sanitize.yaml

# With deterministic seed
firestore2csv export -p myproject --sanitize sanitize.yaml --seed 42
```

The `--sanitize` flag is polymorphic: if the value ends in `.yaml` or `.yml`, it is loaded as a config file. Otherwise, it is parsed as inline comma-separated pairs.

The `--seed` flag is only meaningful when `--sanitize` is also provided.

### Sanitize subcommand

```bash
# From directory
firestore2csv sanitize --config sanitize.yaml -i ./export -o ./sanitized

# From single file
firestore2csv sanitize --config sanitize.yaml -i data.csv -o ./sanitized

# Inline config
firestore2csv sanitize --config "email=email,name=firstName" -i ./export -o ./sanitized

# With seed
firestore2csv sanitize --config sanitize.yaml -i ./export -o ./sanitized --seed 42
```

- `--config` is required (same polymorphic parsing as `--sanitize`).
- `--input` / `-i` defaults to `.` (current directory). Accepts files or directories (recursively discovers `.csv` files).
- `--output` / `-o` is required — sanitized files are always written to a separate location to avoid overwriting originals.
- `--seed` defaults to `0` (fully random).

## Architecture

### Sanitization engine (`sanitize.go`)

All sanitization logic lives in a new file `sanitize.go`, completely independent of export/import logic. The core is a `sanitizer` struct:

- Holds the field-name → faker-type mapping.
- Holds a `*gofakeit.Faker` instance (seeded or random).
- Exposes a method to transform a `map[string]any` by replacing matched string field values with fake data.
- Walks nested maps and arrays of maps recursively.

### Integration with export

Sanitization is applied as a transformation layer on `docRecord.data` **after** reading documents from Firestore but **before** passing them to `writeCollectionCSV`. The export pipeline itself is not modified — the sanitizer simply mutates the data map in place.

### Sanitize subcommand

The `sanitize` subcommand is a pure CSV-to-CSV pipeline:

1. Discover CSV files in the input path (reuse the same directory-walking logic as `import`).
2. Read each CSV file.
3. For each data row, apply the sanitizer to cell values in columns that match the config.
4. Write the sanitized CSV to the output directory, preserving the relative directory structure.
5. Special columns (`__path__`, `__fs_types__`) are never sanitized.

### Config parsing

A shared function parses the sanitization config from either format:
- If the string ends in `.yaml`/`.yml` → read and unmarshal the file.
- Otherwise → split on commas, split each pair on `=`, build the field map.
- Validate that all faker types are recognized; return an error for unknown types.
